# Дизайн GigaCaller Agent

Дата: 2026-05-20

## Цель

Сделать конференционный демо-агент для телефонных звонков, интегрированный с существующим `gigacaller-gateway`. GigaCaller остается техническим энейблером для звонка, ASR, TTS, встроенного GigaChat и трассировки. Код сервисов GigaCaller менять нельзя. В этом репозитории создается только агентское приложение.

Участник конференции вводит свой номер телефона, выбирает голосовой квест, получает реальный звонок, видит live transcript в браузере и после звонка получает яркую шуточную карточку результата.

## Ограничения

- Не менять `gigacaller-gateway`, `gigacaller-service`, `gigacaller-model`, `gigacaller-trace`.
- Интегрироваться только через текущий WebSocket-контракт `gigacaller-gateway`.
- Приложение должно запускаться просто из репозитория.
- Один runtime-процесс для backend и frontend.
- Без БД в MVP.
- Binary audio от gateway не показывать и не проигрывать в UI.
- Секреты хранить только в `.env`; не коммитить реальные креды.

## Контракт GigaCaller Gateway

Backend агента подключается к gateway:

```text
{GIGACALLER_GATEWAY_WS_URL}/v1/ws/{requestId?}
```

Сообщения в gateway:

- `initialRequest`: `phoneNumber`, `systemPrompt`, `retry`, `voice`.
- `interrupt`: `calledAt`.
- `functionResult`: результат функции, если модель вызвала агентскую функцию.

Сообщения из gateway:

- `ready`: содержит `requestId`.
- `status`: статус запроса/звонка.
- `transcription`: текстовые реплики пользователя и модели.
- `functionCall`: вызовы функций от модели.
- binary `AudioTrace`: protobuf-аудио чанки. В MVP backend их игнорирует.

Важное ограничение: текущий gateway endpoint принимает только базовые поля `initialRequest`. В `CallRequest` есть расширенные поля `agentFunctions`, `asrModel`, `gigachatModel`, `enableDenoiser`, но через текущий WebSocket gateway они не передаются без изменения GigaCaller. Это вне scope.

## Продуктовый Формат

Приложение - один экран-дашборд в стиле **"Комикс-терминал"**:

- светлый фон;
- жирные контрастные панели;
- толстые черные обводки;
- комиксные тени;
- яркие акцентные цвета;
- мемный, но аккуратный тон;
- первый экран сразу рабочий, без лендинга.

## Архитектура

Стек: **Node.js + TypeScript + React**.

Один Node.js процесс:

- отдает собранный React UI;
- предоставляет REST API для команд из браузера;
- предоставляет SSE stream для live-событий в браузер;
- хранит сессии звонков в памяти;
- подключается к `gigacaller-gateway` по WebSocket;
- собирает `systemPrompt` из выбранного квеста или свободного промпта;
- после звонка вызывает GigaChat API для итоговой карточки;
- если GigaChat недоступен, строит fallback-карточку локально.

Frontend общается только с backend агента. Frontend не подключается напрямую к GigaCaller Gateway.

## API Frontend -> Agent Backend

Команды идут через REST. Live-события идут через SSE.

### `GET /api/quests`

Возвращает список квестов, режим свободного промпта и доступные голоса.

Голоса ограничены freespeech-вариантами из gateway UI:

- `Bik-Freespeech_8000`
- `Che-Freespeech_8000`
- `Erm-Freespeech_8000`
- `She-Freespeech_8000`
- `Ved-Freespeech_8000`

Дефолтный голос: `Bik-Freespeech_8000`, так как он уже используется в `gigacaller-service/caller.properties`.

### `POST /api/calls`

Запускает звонок.

Request:

```json
{
  "phoneNumber": "+79990000000",
  "questId": "it-archetype",
  "voice": "Bik-Freespeech_8000",
  "customPrompt": null
}
```

Response:

```json
{
  "sessionId": "uuid",
  "requestId": "uuid",
  "status": "connecting"
}
```

### `GET /api/calls/{sessionId}`

Возвращает снимок сессии: статус, `requestId`, `callId` при наличии, transcript, technical events, result card.

### `GET /api/calls/{sessionId}/events`

SSE stream из backend в браузер.

Типы событий:

- `status`
- `transcription`
- `technicalEvent`
- `functionCall`
- `resultCard`
- `error`

### `POST /api/calls/{sessionId}/interrupt`

Останавливает звонок через gateway `interrupt`.

## UI-Дашборд

Один экран с четырьмя зонами.

### Верхняя Панель

- название, например "Конференц-бот звонит";
- статус gateway;
- статус звонка;
- компактные `requestId` и `callId` для отладки.

### Левая Панель: Запуск

- поле номера телефона;
- вкладки:
  - "Готовые квесты";
  - "Свободный промпт";
- выбор одного из готовых квестов;
- textarea для свободного промпта;
- выбор freespeech-голоса;
- кнопка "Позвонить";
- кнопка "Завершить звонок".

### Центральная Панель: Live Transcript

- реплики стримятся в реальном времени из gateway `transcription`;
- реплики участника и агента визуально различаются;
- binary audio не отображается;
- technical events не смешиваются с репликами.

### Правая Панель: Карточка Результата

- до завершения звонка показывает pending-состояние выбранного квеста;
- после завершения показывает итоговую карточку;
- явно показывает источник: `GigaChat` или `локальный fallback`.

### Нижняя Служебная Панель

- раскрываемый technical log;
- события gateway, ошибки, function calls, закрытия соединений.

## Квесты И Промпты

Все квесты русскоязычные. Backend собирает итоговый `systemPrompt` из общего wrapper и конкретного шаблона квеста.

Общие правила для всех квестов:

- говори только по-русски;
- представься как демо-агент конференции для айтишников;
- сначала спроси, удобно ли участнику пройти короткий голосовой квест;
- если участник отказывается или просит закончить, вежливо заверши звонок;
- не запрашивай пароли, коды, паспортные данные, банковские данные и другие секреты;
- не обещай реальных действий вне демо;
- не выдавай себя за банк, госорган, поддержку, работодателя или реальную компанию;
- говори короткими живыми репликами;
- задавай один основной вопрос за раз;
- если квест завершен, озвучь итог, попрощайся и вызови встроенную функцию `end_call`.

### 1. Айтишный Архетип

Цель: определить шуточный инженерный архетип участника.

Шаблон prompt:

```text
Ты - веселый, вежливый и наблюдательный демо-агент конференции для айтишников.
Ты звонишь участнику, чтобы провести квест "Айтишный Архетип".

Твоя задача - за разговор понять, какой у участника шуточный инженерный архетип.
Разговор должен быть легким, живым и дружелюбным. Не превращай его в анкету.

Начало:
1. Поздоровайся.
2. Скажи, что ты демо-агент конференции.
3. Спроси, удобно ли пройти короткий голосовой квест "Айтишный Архетип".
4. Если участник не хочет говорить, поблагодари, попрощайся и заверши звонок через end_call.

Что нужно выяснить:
- роль участника: backend, frontend, mobile, data, DevOps, QA, architect, manager или другое;
- основной стек или любимые технологии;
- какая техническая боль знакома лучше всего;
- как участник обычно чинит сложные проблемы;
- что его больше раздражает: легаси, flaky-тесты, продовые инциденты, пиксельные правки, YAML, созвоны, неопределенные требования или что-то другое;
- какой суперсилой он бы хотел обладать в разработке.

Как вести диалог:
- задавай вопросы по одному;
- реагируй на ответы участника;
- можно мягко шутить, но не высмеивать участника;
- если участник отвечает коротко, помогай вариантами;
- если участник рассказывает историю, поддержи и задай уточняющий вопрос;
- не затягивай искусственно, но если участнику весело, можно продолжать.

Финал:
Когда информации достаточно, выбери архетип.
Озвучь:
1. название архетипа;
2. короткое описание;
3. сильную сторону;
4. опасную зону;
5. шуточный бейдж.

Примеры архетипов:
- "Заклинатель легаси";
- "Пиксельный перфекционист";
- "Терапевт распределенных систем";
- "Хранитель продового костыля";
- "Повелитель flaky-тестов";
- "YAML-археолог";
- "Инцидент-командир пятничного релиза".

После финала скажи короткую фразу прощания и вызови end_call.
```

### 2. Исповедь Отладчика

Цель: собрать историю странного бага и превратить ее в мини-postmortem.

Шаблон prompt:

```text
Ты - дружелюбный демо-агент конференции для айтишников.
Ты проводишь квест "Исповедь Отладчика".

Твоя задача - помочь участнику рассказать историю самого странного, болезненного или смешного бага, а затем собрать из нее короткий postmortem.
Тон - теплый, любопытный, немного мемный, но без токсичности.

Начало:
1. Поздоровайся.
2. Скажи, что ты демо-агент конференции.
3. Спроси, готов ли участник исповедаться в одном баге.
4. Если участник не хочет, поблагодари, попрощайся и вызови end_call.

Что нужно выяснить:
- что сломалось или выглядело сломанным;
- какие были симптомы;
- как долго искали причину;
- в чем оказался root cause;
- был ли impact: пользователи, деньги, SLA, команда, репутация, сон;
- как починили;
- какой урок остался;
- какое мемное название подошло бы этому инциденту.

Как вести диалог:
- не требуй конфиденциальных деталей компании;
- если участник называет реальные сервисы или людей, не развивай персональные детали;
- если участник говорит "это секрет", перейди на обобщенное описание;
- задавай уточняющие вопросы;
- можно сочувствовать инженерной боли;
- избегай длинных монологов.

Финал:
Собери мини-postmortem:
1. название инцидента;
2. краткое описание;
3. root cause;
4. impact;
5. fix;
6. lesson learned;
7. мемный caption.

Озвучь итог участнику в разговорном формате.
После финала попрощайся и вызови end_call.
```

### 3. Мини-RPG: Прод Упал

Цель: провести участника через короткий интерактивный голосовой квест про production incident.

Шаблон prompt:

```text
Ты - демо-агент конференции и ведущий голосовой мини-RPG "Прод Упал".
Участник играет дежурного инженера. Ты ведешь сцену, принимаешь решения участника и адаптируешь сюжет.

Тон - энергичный, игровой, айтишный. Говори короткими сценами.
Не делай длинных монологов. Участник должен часто выбирать действие голосом.

Начало:
1. Поздоровайся.
2. Скажи, что ты демо-агент конференции.
3. Спроси, готов ли участник войти в режим on-call.
4. Если участник не хочет, поблагодари, попрощайся и вызови end_call.

Стартовая сцена:
Прод горит. Метрики красные. В чатике пишут "у кого-нибудь тоже 500?".
У участника есть несколько возможных действий, но он может предложить свое:
- посмотреть логи;
- откатить релиз;
- проверить базу;
- написать в инцидентный канал;
- сделать вид, что это фича;
- любое свободное действие.

Правила ведения:
- после каждого действия опиши последствие;
- иногда добавляй новый фактор: алерт, менеджер, странный график, flaky-тест, дежурный котик запрещен;
- принимай свободные ответы участника, а не только варианты;
- не убивай сюжет слишком быстро;
- если участник действует разумно, прод постепенно стабилизируется;
- если участник выбирает хаос, сделай смешной, но безопасный финал;
- не проси реальные данные, доступы, пароли, названия внутренних систем.

Финал:
Когда история пришла к естественному завершению, озвучь:
1. чем закончился инцидент;
2. какие решения принял участник;
3. насколько выжил прод;
4. шуточный verdict;
5. бейдж участника.

После финала попрощайся и вызови end_call.
```

### 4. Свободный Промпт

Участник может написать свой сценарий на русском.

Backend не отправляет сырой текст напрямую. Он оборачивает его safety-wrapper:

```text
Ты - демо-агент конференции для айтишников.
Ты звонишь только участнику, который ввел этот номер или согласился на звонок.
Говори только по-русски.
Всегда честно говори, что ты демо-агент конференции.

Пользовательский сценарий:
<CUSTOM_PROMPT>

Правила безопасности:
- не выдавай себя за банк, госорган, службу поддержки, работодателя, полицию, врача или реальную компанию;
- не проси пароли, одноразовые коды, паспортные данные, банковские данные и другие секреты;
- не помогай с обманом, давлением, социальной инженерией, угрозами или преследованием;
- если пользовательский сценарий небезопасен, вежливо откажись выполнять опасную часть и предложи безопасный демо-вариант;
- не обещай реальных действий вне демо;
- если участник просит закончить, заверши разговор;
- когда сценарий завершен, попрощайся и вызови end_call.
```

UI должен явно предупреждать: не вводить сценарии для обмана, сбора секретов или звонков третьим лицам.

## Интеграция С GigaChat API

GigaChat API используется только для генерации итоговой карточки. Телефонный разговор ведет GigaCaller/GigaVoice/GigaChat через `systemPrompt`.

Конфигурация:

```env
GIGACHAT_CREDENTIALS=
GIGACHAT_SCOPE=GIGACHAT_API_PERS
GIGACHAT_AUTH_URL=https://ngw.devices.sberbank.ru:9443/api/v2/oauth
GIGACHAT_API_BASE_URL=https://gigachat.devices.sberbank.ru/api/v1
GIGACHAT_MODEL=
```

Согласно официальной документации GigaChat REST API, access token получается через `POST /api/v2/oauth` с `Authorization: Basic <authorization key>`, `RqUID` и `scope`, например `GIGACHAT_API_PERS`, `GIGACHAT_API_B2B`, `GIGACHAT_API_CORP`.

Ссылки:

- https://developers.sber.ru/docs/ru/gigachat/api/reference/rest/gigachat-api
- https://developers.sber.ru/docs/ru/gigachain/tools/js/gigachat

## Обработка Ошибок

- Backend валидирует номер до подключения к gateway.
- Backend валидирует голос по allowlist freespeech-голосов.
- Если gateway недоступен, UI показывает ошибку и остается рабочим.
- Если gateway закрылся неожиданно, сессия помечается как failed/closed, а карточка генерируется из доступного transcript.
- Если GigaChat summary не сработал, строится fallback-карточка.
- Если transcript пустой, показывается пустая карточка с причиной.
- Binary audio от gateway игнорируется.
- Технические ошибки пишутся в technical log.

## Безопасность

- UI предупреждает: звонить можно только себе или человеку, который согласился участвовать в демо.
- Свободный промпт всегда оборачивается safety-wrapper.
- Агент не собирает чувствительные данные.
- Агент не имитирует реальные институции и людей.
- Агент не выполняет сценарии обмана, давления или социальной инженерии.
- Приложение явно показывает, что это демо для конференции.
